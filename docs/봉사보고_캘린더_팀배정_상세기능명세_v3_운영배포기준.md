## 4) 데이터 정합성 정책 (핵심)

## 4.1 중복 신청 방지 (필수)

중복 신청은 프론트 검증만으로는 막을 수 없으므로 DB 제약이 최종 기준이다.

1. 프론트 사전 검증
- 같은 화면에서 동일 슬롯 중복 클릭 방지(disabled/loading)
- 이미 신청한 슬롯은 버튼 비활성화

2. 서버 검증
- 신청 요청 시 아래 확인:
  - 유효한 사용자/권한
  - 슬롯 상태(`open`) 여부
  - 동일 사용자 동일 세션 중복 여부

3. DB 제약 (최종 방어)
- 예시 고유키:
  - `UNIQUE(event_id, session_key, applicant_user_id)`
  - 필요 시 `UNIQUE(slot_id, applicant_user_id)`

4. 충돌 응답
- 중복 시 `409 Conflict`
- 메시지: “이미 해당 시간대에 신청되어 있습니다.”

## 4.2 중복 배정 방지 (필수)

1. 규칙
- 동일 사용자 동일 시간대 다중 팀 배정 금지

2. DB 제약
- `UNIQUE(event_id, assignee_user_id)`

3. 배정 전환
- 기존 배정이 있으면 `이동` 처리(삭제 후 추가 또는 update)

## 4.3 Timestamp 사용 원칙

결론: **timestamp만으로 중복 방지를 하지 않는다.**

1. timestamp 용도
- 정렬(createdAt)
- 감사로그(updatedAt)
- 충돌 표시(마지막 수정 시각)

2. 중복 방지 핵심
- 고유키 + 트랜잭션 + 서버 검증

3. 권장 보강
- `idempotencyKey`(UUID) 추가 가능:
  - 네트워크 재시도 시 동일 요청 중복삽입 방지

## 5) 트랜잭션 처리 규칙 (신청/배정)

## 5.1 신청 트랜잭션

1. 요청 수신
2. `prisma.$transaction` 시작
3. 대상 슬롯 조회 + 상태 확인
4. 중복 신청 여부 조회
5. 신청 row insert
6. 필요 시 슬롯 상태 부분 갱신
7. 커밋 후 `revalidatePath`

## 5.2 관리자 배정 트랜잭션

1. 배정 대상 슬롯/팀/사용자 조회
2. 중복 배정 여부 조회
3. 배정 insert/update
4. 팀 인원수 규칙 검사(2~3명 권장)
5. 커밋 후 캘린더/상세 경로 재검증

## 6) 실시간 반영 전략 (운영 기준)

## 6.1 Phase A (즉시 적용, 안정 우선)

1. 변경 트리거
- 모든 신청/배정/보고 액션 후 `revalidatePath`

2. 클라이언트 반영
- 작성자: 즉시 `router.refresh()`
- 다른 사용자: 10~20초 polling(가벼운 summary API)

장점: 구현 단순, Vercel 환경에서 안정적

## 6.2 Phase B (고도화, 선택)

1. Supabase Realtime 채널 도입
- 테이블 변경 이벤트 구독
- 변경된 날짜/이벤트만 부분 갱신

2. 폴백
- Realtime 연결 실패 시 polling 자동 전환

## 7) Prisma 스키마 확장 확정안(v3)

신규 모델(명칭 예시):

1. `ScheduleEvent`
- `id`, `date`, `title`, `sessionKey(AM|PM)`, `timeLabel`, `zone`, `leaderName`, `guideText`

2. `ScheduleSlot`
- `id`, `eventId`, `slotNo`, `status(open|assigned|closed)`, `label`

- `id`, `eventId`, `slotId`, `applicantUserId`, `applicantNameSnapshot`, `note`, `createdAt`

5. `TeamAssignment`
- `id`, `eventId`, `teamId`, `assigneeUserId`, `assigneeNameSnapshot`, `createdAt`
- unique: `(eventId, assigneeUserId)` (동일 이벤트 중복 배정 방지)

6. `ServiceReport`
- `id`, `reportDate`, `reporterUserId?`, `reporterName`, `status(draft|submitted)`
- `statsJson`, `subwayJson`, `experienceJson`, `notes`, `createdAt`, `updatedAt`
- index: `(reportDate)`, `(reporterName)`

## 8) API/Action 계약 (운영형)

## 8.1 조회

1. `getScheduleMonthAction(month: "YYYY-MM")`
2. `getScheduleDateAction(date: "YYYY-MM-DD")`
3. `getEventTeamsAction(eventId: string)`
4. `getMyReportsAction()`
5. `getAdminReportsAction(filters)`

## 8.2 변경

1. `applySlotAction(formData)`
- 입력: `eventId`, `slotId`, `note?`
- 성공: `{ ok: true }`
- 실패:
  - `409` 중복 신청
  - `403` 권한
  - `422` 유효성

2. `assignTeamAction(formData)` (ADMIN/LEADER)
- 입력: `eventId`, `teamId`, `userId`
- 실패: `409` 중복 배정

3. `submitReportAction(formData)`
- draft/submitted 상태 분리

4. 보고서 제출은 로그인 사용자 전용(비회원 제출 미지원)

## 14) 테스트 시나리오 (배포 전 필수)

1. 정상 시나리오
- 일반 사용자가 신청 성공
- 관리자가 팀 배정 성공
- 보고 제출 성공

2. 충돌 시나리오
- 동일 사용자가 같은 시간대 2회 신청 → 2번째 409
- 동일 사용자를 2팀에 배정 시도 → 409

3. 권한 시나리오
- USER가 관리자 액션 호출 → 403

4. 실시간 시나리오
- 사용자A 신청 후 사용자B 화면 갱신(Phase A polling 기준 20초 내 반영)
